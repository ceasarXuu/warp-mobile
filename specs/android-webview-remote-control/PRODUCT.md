# Android WebView Remote Control Product Spec

## 背景

当前项目已支持桌面端和 Web 远控分享链接。移动端第一阶段不重写远控渲染和会话协议，而是把已有 Web 远控页面封装进 Android 原生应用，由原生层补齐移动端必须具备的能力，尤其是终端友好的内置键盘。

## 产品目标

1. 用户在 Android 上打开 Warp 远控分享链接后，可以进入一个稳定、全屏、可恢复的移动远控体验。
2. 移动端保留 Web 远控页面的现有会话能力，不引入新的会话协议分叉。
3. Android 原生层提供终端键盘、链接接管、会话状态提示、错误恢复和诊断日志。
4. 内置键盘复用 Astropath 的核心设计：系统输入法和内置键盘双模式、三段键盘布局、修饰键状态机、长按重复、反馈和更多按键面板。

## 非目标

- 第一阶段不把桌面端完整功能搬到 Android。
- 第一阶段不重写远控 Web/WASM 渲染器。
- 第一阶段不支持创建、管理或分享远控会话，只消费已有分享链接。
- 第一阶段不绕过服务端或 Web 端既有权限校验。

## 目标用户路径

### 打开分享链接

用户从浏览器、聊天软件或系统剪贴板打开 Warp 分享链接。Android 应用识别链接中的 session id 和 query 参数，进入远控页面。若 WebView 加载失败，应用显示可重试、复制链接、在浏览器打开的恢复动作。

### 操作远程终端

用户进入会话后，默认看到远控页面和底部内置键盘。键盘提供 Esc、Tab、Ctrl、Alt、Shift、方向键、常用符号、Backspace、Enter 等终端高频按键。用户可以切换到系统输入法，也可以从系统输入法返回内置键盘。

### 会话状态变化

当用户只是 viewer、会话断开、权限不足、连接中或已结束时，移动端必须明确改变输入能力。键盘不能继续发送不可执行输入，但仍可保留复制链接、重试、查看状态等非输入动作。

### 诊断和反馈

当远控不可用、键盘输入丢失或 WebView 页面异常时，日志可以定位问题发生在链接解析、WebView 加载、JS bridge、WASM 输入、会话角色还是网络连接。

## 核心体验要求

### 移动容器

- App 启动后优先处理分享链接，而不是进入营销页或空首页。
- 支持冷启动、热启动、后台恢复和屏幕旋转。
- 远控页面使用沉浸式布局，但状态栏、导航栏和键盘区域不能遮挡终端关键内容。
- WebView 的进度、错误、证书和重定向策略必须可观测。

### 内置键盘

- 默认模式为内置键盘，保留切换到系统输入法的入口。
- 内置键盘采用 Astropath 的三段模型：
  - 左侧窥视区放启动类和补充符号类按键。
  - 中心区放主键盘、修饰键、空格、回车和退格。
  - 右侧窥视区放导航键、管道、重定向、括号等终端符号。
- 支持拖拽切换左侧、中心、右侧三个锚点。
- Ctrl、Alt、Shift 支持 inactive、one-shot、locked 三态。
- Backspace、Delete、方向键等可长按重复。
- 所有按键必须通过统一动作模型发送，不能在多个入口里各自拼接协议。

### 权限和安全

- WebView 只允许加载 Warp 认可的远控域名。
- JS bridge 必须校验来源、消息版本和事件类型。
- 日志不得记录完整分享链接、token、cookie、用户输入明文。
- 输入能力由会话角色和连接状态决定，Android 键盘不能自行假设可执行。

## 验收标准

1. Android 安装包能通过分享链接直接进入指定远控会话。
2. 同一分享链接在浏览器 Web 和 Android WebView 中进入同一会话，不产生不同会话语义。
3. 内置键盘可发送 printable、Esc、Tab、Enter、Backspace、Delete、方向键、Home、End、PageUp、PageDown、Ctrl+C、Ctrl+D、Ctrl+Z、Alt 前缀输入。
4. 系统输入法和内置键盘可以互相切换，切换后焦点仍回到远控终端。
5. 断线、权限不足、viewer-only、WebView 加载失败都有明确 UI 状态和日志。
6. 冒烟测试覆盖链接打开、页面加载、键盘输入、权限禁用、后台恢复。
7. 关键日志可以按 session id 哈希、事件序列号和 bridge 消息 id 串起端到端路径。
